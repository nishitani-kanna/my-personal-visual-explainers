# よく図解するテーマ・用語変換

## このスキルでよく図解するテーマ

### 1. セキュリティリテラシー（最頻出）

**背景**: AIツールの業務利用が広がる中、セキュリティ意識が人によって大きく異なる。
個人情報・業務機密情報の取り扱い、社内ガイドラインの理解が課題。

| 図解テーマ例 | 一言サマリーの方向性 |
|------------|-----------------|
| AIへの情報入力ルール | 「入力前に『社外に出していい？』の一問」 |
| 個人情報とは何か | 「特定の人を識別できる情報はすべて個人情報」 |
| フィッシング詐欺の見分け方 | 「URLと送信元ドメインの確認が第一歩」 |
| パスワード管理 | 「使い回しゼロ＋パスワードマネージャーで解決」 |
| SNSでの情報発信リスク | 「仕事のことをSNSに書く前に一呼吸」 |
| 社内情報の持ち出しルール | 「私用PCや個人クラウドに業務データを保存しない」 |

### 2. 開発基礎（研修でよく出るつまずきポイント）

| 図解テーマ例 | 一言サマリーの方向性 |
|------------|-----------------|
| Gitの基本（add/commit/push） | 「セーブ・コメント付きセーブ・アップロードの3ステップ」 |
| ブランチとは | 「本番を壊さずに試せる『作業コピー』」 |
| プルリクエストとは | 「変更を本番に取り込む前の確認依頼」 |
| エラーの読み方 | 「怖くない。エラー文は『何が・どこで・なぜ』を教えてくれている」 |
| APIとは | 「ソフトウェア同士が会話するための窓口」 |
| データベースの基本 | 「表形式でデータを保存する引き出し」 |
| 環境構築とは | 「プログラムを動かすための土台づくり」 |
| ログとは | 「プログラムが残す行動記録。デバッグの手がかり」 |

## 用語のやさしい言葉への変換

### セキュリティ系

| 技術用語 | やさしい言葉 | たとえ |
|---------|------------|--------|
| 個人情報 | 特定の人を識別できる情報 | 名前・住所・電話番号など「その人だとわかる情報」 |
| 機密情報 | 社外に出してはいけない情報 | 会議の議事録、設計書、顧客リストなど |
| 情報漏えい | 外部に情報が流れ出ること | 鍵のかかっていない引き出しから書類が持ち出される |
| フィッシング | 偽のサイト・メールで情報を騙し取る詐欺 | 偽の宅配業者が玄関で個人情報を聞き出す |
| 認証 | 本人確認 | パスポートを見せて「本人です」と証明する |
| パスワードマネージャー | パスワードを安全に一括管理するアプリ | 鍵を全部まとめて管理する金庫 |
| 二段階認証 | パスワード＋もう1つの確認 | 銀行カード（持ち物）＋暗証番号（記憶） |
| VPN | 暗号化された専用通信路 | 覗き見されない専用の電話回線 |

### 開発ツール系

| 技術用語 | やさしい言葉 | たとえ |
|---------|------------|--------|
| Git | バージョン管理ツール | 変更履歴を残せるセーブ機能 |
| commit | 変更内容を記録する | コメント付きセーブ |
| push | リモートに送る | データをサーバーにアップロードする |
| pull | リモートから取得する | サーバーの最新データをダウンロードする |
| branch | 作業用コピー | 本番を壊さずに試せる別の作業台 |
| merge | 変更を本番に取り込む | 試作品を正式な製品ラインに統合する |
| プルリクエスト | 変更の確認依頼 | 「このページの修正、レビューをお願いします」という申請 |
| リポジトリ | コードの保管場所 | プロジェクトのデータが入ったフォルダ |
| コンフリクト | 変更の衝突 | 同じ場所を2人が同時に書き直したときの矛盾 |

### プログラミング基礎

| 技術用語 | やさしい言葉 | たとえ |
|---------|------------|--------|
| 変数 | 名前のついた箱 | ラベルを貼った引き出し |
| 関数 | 処理のまとまり | 自動販売機（入力すると出力が返ってくる） |
| 引数 | 関数に渡す情報 | 自販機に入れるコイン |
| 戻り値 | 関数が返す結果 | 自販機から出てくる商品 |
| クラス | 設計図 | 「社員証の雛形」—何の項目があるかを定義する |
| インスタンス | 設計図から作った実体 | 雛形をもとに作った「田中さんの社員証」 |
| API | ソフトウェア同士の窓口 | 銀行の窓口。決まった書類を出すと決まったサービスが受けられる |
| デバッグ | バグを見つけて直す作業 | レシピ通りに作ったのに味がおかしいときの原因探し |
| ログ | プログラムの行動記録 | 工場の作業日報 |
| エラー | 処理の失敗通知 | 「この手順では動けません」というプログラムからのメッセージ |
| 環境 | プログラムが動く場所・条件 | 料理をするための台所（設備が揃っていないと動かない） |
| ライブラリ | 誰かが作った便利な部品集 | 料理レシピ本（1から作らず活用できる） |
| フレームワーク | 開発の骨格・ルールセット | 建物を建てるときの鉄骨構造（中身を入れるだけ） |

## 新人がつまずきやすいパターンと対処

### セキュリティ系でよくある誤解

| 誤解 | 正しい理解 |
|-----|---------|
| 「社内のAIなら何を入力してもいい」 | AIサービスによってはデータが学習に使われる場合がある。社内ガイドライン確認が先 |
| 「名前だけなら個人情報じゃない」 | 他の情報と組み合わせると特定できるので個人情報になる場合がある |
| 「仕事の愚痴をSNSに書いても問題ない」 | 会社・案件・取引先が特定できる内容はNG |
| 「パスワードは長ければ何でもいい」 | 推測されやすい単語の組み合わせは危険。ランダム文字列＋パスワードマネージャーが正解 |

### 開発でよくある誤解

| 誤解 | 正しい理解 |
|-----|---------|
| 「エラーが出たら自分のミス」 | エラーは情報。何が・どこで・なぜおかしいかを教えてくれるもの |
| 「コミットは大きな変更のときだけ」 | 小さな単位でこまめにコミットするほど後で追いやすい |
| 「ブランチはベテランが使うもの」 | 一人でも使う。本番コードを守るためにブランチは基本 |
| 「コードは自分で全部書かないといけない」 | ライブラリやフレームワークを活用するのが現代の開発スタイル |
